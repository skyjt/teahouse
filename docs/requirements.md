## 1. 背景与目标

茶话间是纯内网、无服务器、基于 IP 的局域网 IM + 文件传输工具。设计原则：

1. **零服务器、零配置**：装上即用，自动发现，所有节点对等
2. **纯内网**：数据不出局域网，无外网依赖、无遥测
3. **办公可用**：消息可靠（不静默丢失）、文件传输跑满内网带宽
4. **老设备友好**：Win7 / Debian 10 / macOS 26 起（见 README）

## 2. 目标场景与用户

- **主场景：中大型办公内网**——几十到几百人，多 VLAN/网段，跨网段互通是刚需，存在部门结构。
- 次场景（自然兼容，不专门优化）：小团队单网段、个人多设备互传。
- **规模设计目标：单网络 ≤ 1000 在线节点**（决议 #15，由 500 上调），性能与协议设计按此预算。
- 用户画像：普通办公人员（非技术背景，要求开箱即用）；IT 管理员（负责分发安装、网段配置）。
- 环境事实：内网的 Win7 终端为统一部署的 **64 位虚拟机**（决议 #20 放弃 32 位的背景；VM 弱显卡的渲染对策见 tech-design.md §9）。

## 3. 非目标（Non-Goals）

- 不做互联网/跨公网通信，不做云同步、云存储
- 不做账号体系与中心服务器（包括"轻服务器"）
- 不做移动端（仅三平台桌面）
- 不做富文本消息（加粗/字号等），不做视频通话；语音对讲、远程协助、共享文件柜列为远期评估（决议 #7）
- 不做已读回执，消息状态只到"已送达"（决议 #1）
- 不做传输加密：信任内网边界，沿用 ipmsg / iptux 的明文做法（决议 #5）
- v1 不做 IPMSG/飞秋协议互通：协议**借鉴其机制（广播发现、UDP+ACK、TCP 文件）但报文自有**（决议 #5），兼容模式留远期

## 4. 术语

| 术语 | 含义 |
|---|---|
| 节点（peer） | 一个运行中的茶话间客户端实例 |
| 节点 ID | 首次启动生成的本机唯一标识，身份不随 IP/昵称变化 |
| 会话 | 与某个节点或讨论组的聊天上下文 |
| 讨论组 | 固定成员列表的多人会话（P2P 语义，消息向成员逐个直发） |
| 分组 | 通讯录的"公司 > 部门 > 团队"三级维度，由用户在设置中自报 |
| 补发 | 对方离线时消息存于发送端本地队列，探测到上线后自动重发 |

## 6. 功能需求明细

### 6.1 发现与在线状态

- **F-DISC-1 同网段自动发现（P0）**：启动时 UDP 广播上线，在线节点应答（带随机延迟，避免 1000 节点同时应答造成风暴）；收到上线广播即更新列表。下线发广播，异常退出靠心跳超时判离线。
- **F-DISC-2 跨网段发现（P0，主场景刚需）**，三个手段按序落地：
  1. 手动添加对端 IP / 导入 IP 列表（v0.1，保底手段）；
  2. 指定网段定向单播探测（如 `10.1.0.0/24`，可配多段，限速扫描）；
  3. 节点间在线列表交换（gossip 散播：A 知道的节点告诉 B，跨网段只要有一个"桥"节点即可全网打通——内网通同思路）。
- **F-DISC-2a 扫描范围自动分享（P0，决议 #114）**：用户手动新增的扫描 CIDR 会作为“网段记录”在已在线节点间低频同步；收端只自动加入本机网段记录并标注来源，不立刻全段扫描。自动接收的网段进入后台节流扫描：首次随机延迟 30–90 分钟，同一网段最短 12 小时自动扫一次；在线节点超过 50 时按稳定 hash 抽样，仅约 10% 客户端参与该网段后台扫描，其余节点仍靠既有 `peers` gossip 学到在线节点。用户手动点击“扫描/再次扫描”不受后台节流影响。用户移除某个同步来的网段后，本机记入忽略表，远端再次分享时不自动加回；自己重新手动添加会清除忽略。
- **F-DISC-2b 主界面全局网段刷新（P0，决议 #115）**：主窗口左侧导航栏在设置按钮上方提供“刷新全局用户”按钮。点击后按本机已保存的所有合法扫描网段汇总、去重主机地址，并以手动扫描同等限速逐个单播探测；扫描期间在左侧栏显示进度条，进度含义为“已发出探测地址数 / 总地址数”。该动作由用户显式触发，不受自动后台扫描的 30–90 分钟延迟、12 小时去重或大网抽样约束；运行中重复点击不并发启动第二轮。
- **F-DISC-3 节点身份（P0）**：首次启动生成节点 ID 并持久化；昵称、IP、主机名都可变，**会话与历史跟着节点 ID 走**。同一机器多实例不在 v1 范围。**同一人多台设备 = 多个独立身份**（设备即身份，无服务器下的天然语义，不做账号合并——明示防误解，决议 #22）。
- **F-DISC-4 通讯录（P0）**：显示昵称、IP、平台图标、在线状态；**三级折叠树：公司 ▸ 部门 ▸ 团队 ▸ 成员**（决议 #10，每级显示在线/总数，字段空缺逐级跳过，全空归"未分组"）；组内在线优先、离线灰显；即时搜索过滤（1000 人规模下搜索是必需品）。布局明细见 [ui-design.md](ui-design.md) §4。
- **F-DISC-5 分组/部门（P1）**：无服务器、无组织架构中心——**用户在"我的资料"设置页自填"公司""部门""团队"三个字段**，随在线协议广播，各端据此聚合成树（决议 #8/#10）。IT 预置配置文件（安装目录放 JSON 锁定字段）降为 P2 可选增强。
- **F-DISC-6 头像（P1，决议 #45/#47）**：v1 = 昵称首字彩色色块 + 20 个 Twemoji 本地 SVG 动物图标 + 背景色模板可选（协议里只传头像编号，不传图；图标下标与背景色下标组合编码，兼容既有 #28 编号语义）；自定义图片头像（需在 P2P 网络传播与缓存）为 P2（决议 #2）。
- **F-DISC-7 联系人持久保留与资料刷新（P0）**：出现过的节点持久保留在通讯录（离线灰显**并加"离线"副文字**，不只靠颜色区分，决议 #11/#17，重启不丢）；资料在对方上线、主动变更、心跳资料版本号失配三种时机自动刷新，防"机器没换、用的人换了"（协议机制见 protocol.md §6.2）。不提供手动删除；"清理 90 天未上线"入设置-高级（P2）。
- **F-DISC-8 在线状态二次校验（P0，决议 #16）**：心跳有 90s 窗口期，"显示在线"可能实际已掉线——**打开会话时**向对方单播探活（复用 entry/alive，约 2s 超时即转离线显示）；**发送消息时**以 ACK 为准，未确认不标"已送达"，重试失败自动转离线补发。双保险杜绝"看着在线、消息黑洞"。
- **F-DISC-9 联系人备注名（P1，v0.3，决议 #22/#37/#40）**：联系人资料页与私聊头部资料弹窗均可设本地备注，备注覆盖昵称显示（重名刚需），列表/会话/搜索全部按备注命中；通讯录单击联系人时在右侧展示完整资料页，双击联系人时直接进入该联系人单聊；备注仅存本机、不广播。

### 6.2 消息

- **F-MSG-1 单聊文本（P0，决议 #46）**：UTF-8 + emoji；短消息优先走 UDP+ACK，UDP 重试耗尽后自动用现有 TCP 控制帧兜底一次；超长消息直接走 TCP（阈值由协议定）；每条消息有全局唯一 ID（去重、补发、撤回的基础）。发送状态三态：发送中 / 已送达 / 失败（可点重发）；不做已读回执（决议 #1）。
- **F-MSG-2 离线暂存补发（P0）**：目标离线时消息入发送端本地持久化队列，UI 标"等待对方上线"；探测到目标上线后按原顺序自动补发；收端按消息 ID 去重。队列默认保留 7 天、单节点上限 200 条，超限提示（决议 #6）。
- **F-MSG-3 图片消息（P0）**：剪贴板粘贴（Ctrl+V 发截图）、拖图、选文件三种入口；UI 内显示缩略图、点击看原图；传输复用文件通道，对端自动接收入图片缓存（不走文件确认流程，有大小上限，超限转普通文件发送）。
- **F-MSG-4 讨论组/群聊（P0）**：任意节点可建组（组 ID + 名称 + 成员列表）；文本消息向成员逐个直发，离线成员走补发；成员可增删，组元数据随消息携带版本号同步。**一致性按"尽力而为"设计**（P2P 无仲裁，接受短暂的成员视图不一致），上限 50 人/组。支持**群内 @ 成员**（P1，被 @ 者通知加强）；成员可自行退组（决议 #22）。建群流程先做联系人快速搜索与多选，再进入下一步设置组名/管理密码；建群时记录创建 IP，并可选填管理密码：密码需二次确认，可填写密码提示（随群元数据同步与备份，明文密码不入库不入协议）；有密码的组允许成员输入密码后改名/增删成员；无密码的组仅创建 IP 可执行群管理（决议 #27/#30）。群聊图片/文件首版复用单聊文件通道，发送端按在线群成员逐个发 `file-ctl offer`，离线成员不入队，收端按 offer 中的 `groupId/groupRev` 入群会话（决议 #32）；群聊图片仅当单图 ≤10MB 时按图片消息发送，超过 10MB 自动退化为普通文件卡片，由接收者手动接收后才开始 TCP 拉取（决议 #33）。
- **F-MSG-5 历史与搜索（P0）**：本地 SQLite 存储，按会话无限滚动。切换不同会话时，消息区必须按会话恢复离开时的滚动位置；若该会话此前停在底部，则回来时仍贴到底部以包含新消息。首次打开一个会话、从通知/托盘/震动等后台入口直达会话、或主窗后台恢复后重新点开会话时，默认定位到最新消息（决议 #111）。**全局搜索（P1，决议 #12）**：分类返回联系人（全量含离线）/讨论组/聊天记录/文件名，命中聊天记录可跳转会话并定位高亮（参考微信，见 ui-design.md §6）。**会话内历史搜索（P1，决议 #34/#35/#36）**：群聊与私聊输入区工具条最右侧提供历史搜索入口，点击弹出设置页尺度的当前会话搜索面板，默认显示最近记录，内置图片、文件、连续日期范围筛选；图片结果直接展示缩略图，结果点击后跳转并高亮对应消息。**导出（P1）**：按会话 + 时间范围，阅读格式 HTML/TXT + **迁移备份包**（消息库+媒体，供导入用）；**导入（P1，决议 #19）**：仅认备份包，导入时旧机器的"我"自动映射为本机身份，与现有记录按消息 ID 去重合并——**换电脑不丢数据**；清空记录带双重确认。
- **F-MSG-6 撤回（P1，v0.4+）**：自己的文本 / 群文本 / PK 消息 **5 分钟内**可撤回（决议 #63/#139，原 2 分钟）；收发两端均隐藏原消息并插入系统提示行。撤回 PK 只影响被撤回的那一条 PK 消息，不级联撤回之后别人另发的 PK 消息。对端离线或暂时不可达时，撤回指令随补发机制送达。图片、文件、表情撤回需先让 `file-ctl` 具备跨端一致消息 ID，留后续扩展。
- **F-MSG-7 表情（P0/P1，决议 #18/#45）**：emoji 面板为 P0（v0.2），但面板与消息正文显示不得依赖系统彩色 emoji 字体；Win7 / UOS 等缺字平台使用项目内本地 SVG 兼容渲染，协议与剪贴板仍保留原始 UTF-8 emoji 字符。**自定义表情包收藏（P1，v0.3）**：聊天中的图片右键"添加到表情"，保存时**自动压缩**（静图最长边 512px 转 WebP，GIF 动图 ≤2MB 原样保留）；表情面板内可删除/排序；发送表情包为轻量图片消息（协议 `kind:"sticker"`），对端免确认接收，气泡内固定小尺寸展示。
- **F-MSG-8 转发与引用（决议 #22）**：消息**转发**到其他会话（文本/图片/文件，P1，v0.4）；**引用回复**（P2）。
- **F-MSG-9 私聊窗口震动（P1，决议 #109/#110/#112）**：仅单聊可用，作为可靠即时提醒动作。震动走 `msg(kind:"nudge")` 投递，有 ACK 才算发送成功；对端离线或无响应时直接失败，不进入离线补发队列，避免上线后补发旧震动造成骚扰。发送成功后，本端插入“你发送了一次窗口震动”本地系统提示行；收端收到且未被限流时，唤起/震动主窗口、自动切到发起人的单聊，并插入“对方发来一次窗口震动”本地系统提示行；若该单聊已设免打扰，收端仍可保留系统提示行，但不得唤起、置前或震动窗口。系统提示行与撤回提示同样居中展示，不参与全文搜索。发送端与接收端均按“同一对端 60 秒最多 2 次、任意两次至少间隔 15 秒”限流；被限流的震动静默丢弃或提示稍后再试，不影响普通消息收发。
- **F-MSG-10 PK 分歧解决（P1，v0.22，决议 #139）**：聊天输入区在表情按钮后、窗口震动前新增小图标按钮 `PK`，图标语义为斜体 `PK` 字标。点击后弹出小浮层，两个并排玩法按钮，图标在上、文字在下，文字为「猜拳」和「骰子」。用户点玩法后才由主进程在发送瞬间随机生成结果并发出结构化 `msg(kind:"pk")`，发送前不展示也不允许用户预选结果；渲染层只负责把已生成结果播放成约 **1.5 秒**的“像随机 GIF 一样”的本地动画。单聊与群聊均可用，但 PK 定义为在线小娱乐：单聊仅对方在线时可发；群聊仅至少一位其他成员在线时可发，并且只投递给当时在线的成员，离线成员不入队、不补发。若发送途中对端掉线，消息标失败，可点重试，但重试沿用同一条消息和同一个已生成结果，不重新随机。每次参与都发送一条新的独立 PK 消息，不修改原消息、不做回合聚合、不输出谁赢谁输，让用户自己看结果。收到他人的 PK 消息时，在气泡旁显示小按钮：猜拳为「我也来」，骰子为「掷一下」；自己的 PK 消息旁不显示参与按钮。允许反复参与，不做每人一次限制或额外防刷屏。骰子结果为 1–6 点，UI 使用真实点数组合的自绘骰面；猜拳结果为 `rock/paper/scissors`，UI 文案显示「出了石头 / 剪刀 / 布」，图形使用本地 Twemoji 原色手势资源（缺资源时补本地 SVG 与署名，禁止 CDN / 远程图片）。新发 / 新收 PK 播放一次动画，历史加载、翻页和重开会话直接显示最终结果。PK 参与普通未读与系统通知，但通知、会话列表预览和历史搜索结果只显示「PK · 猜拳 / PK · 骰子」或「[PK] 猜拳 / [PK] 骰子」，不提前暴露结果；导出 HTML/TXT 可展示完整结果。PK 可撤回，但不可复制、不可转发。旧版本客户端不做 fallback 文本，混用时按未知消息能力处理，要求双方升级后才能看到 PK。该能力是内网轻娱乐，不提供防作弊的承诺揭示、签名或加密公平性证明。
- 链接自动识别可点击（P1）；"正在输入"状态（P2）。

### 6.3 内置截图

- **F-CAP-1 框选截图（P0）**：全局热键（默认 `Ctrl/Cmd+Alt+A`，可改）→ 全屏蒙层框选 → 进剪贴板并可一键发送到当前会话。
- **F-CAP-2 简单标注（P1）**：矩形、箭头、文字、马赛克。
- 平台注意：macOS 需引导"屏幕录制"权限；Windows 注意多屏与高 DPI；Linux 仅承诺 X11（Debian 10 默认），Wayland 下降级提示用系统截图后粘贴。

### 6.4 文件传输

- **F-FILE-1 文件/多文件（P0）**：拖拽/选择/粘贴入口；显示进度、速率、剩余时间；可取消；并发传输队列（默认并发 3，可配）。
- **F-FILE-2 文件夹（P0）**：递归整夹发送，保留目录结构。
- **F-FILE-3 接收策略（P0，决议 #174/#179）**：普通文件默认仍需接收方点击「接收」后开始传输；接收方点击「接收」或私聊「直接发送」自动接收时，若未选择「另存为」，统一保存到 `文件保存位置/联系人名称/`。联系人目录名以本地备注优先、其次对方昵称生成，并做文件名清洗；重名仍自动加后缀，不覆盖。**私聊**新增发送方在已发送文件卡片上触发的「直接发送」能力：发送者先普通发送文件/文件夹，文件卡片出现后，若对方在线且双方版本支持，可点击「直接发送」请求接收方免确认拉取。接收方设置提供总开关「允许同事直接发送文件」，默认开启，关闭后直接发送退化为普通文件卡片。**群聊文件不允许直接发送**，只能走现有文件卡片手动接收；群聊手动接收同样按发送人显示名进入联系人子目录；离线文件仍不入队。
- **F-FILE-4 断点续传（P1，v0.4+）**：分块 + 校验，中断后从断点继续。
- **F-FILE-5 对方离线时发文件**：v1 行为 = 提示对方不在线、不入队（文件在磁盘上随时会变，离线暂存风险高）；群聊文件/图片同样只投递给当前在线成员，离线成员不会进入补发队列；"上线后提醒我重发"作为 P2（决议 #4/#32）。
- **F-FILE-6 传输记录（P1）**：成功/失败/取消的历史，可打开所在目录。

### 6.5 广播与公告

- 中大型内网场景常见诉求（内网通的核心卖点之一），但"人人可全员广播"必被滥用，无服务器下又没有权限中心。**v1 仅做"多选收件人群发"（P1，v0.3，发出后各自成单聊上下文）**（决议 #62：v0.5.26 起取消该入口，改用讨论组替代）；正式的"公告"（置顶、强提醒、全员/分组范围、发布权限）列 P2，等权限/预置配置机制想清楚再做（决议 #3）。

### 6.6 系统集成

- **F-SYS-1 托盘 / 菜单栏（P0，决议 #42）**：常驻托盘；关窗=最小化到托盘。未读总数变化时跨平台刷新系统提示：macOS 菜单栏图标旁显示未读数字并同步 Dock 角标；Windows 使用任务栏 overlay 数字并闪烁托盘图标；Linux 尝试系统角标（Unity 等支持时生效），并以托盘图标闪烁兜底。所有平台 tooltip 都显示未读总数，未读清零后数字与闪烁全部清除。
- **F-SYS-2 桌面通知（P0）**：横幅显示发信人与摘要；点击直达会话；全局免打扰与按会话免打扰（P1）。按会话免打扰不弹系统通知、不触发震动前台动作，会话列表仍累计未读但未读数字显示为灰色；检测到投屏/演示场景自动勿扰（P2，决议 #22）。
- **F-SYS-3 开机自启（P0）**：默认开启（办公场景装完即长驻），安装器与设置中均可关。
- **F-SYS-4 全局快捷键（P1）**：截图、显示/隐藏主窗，可自定义、可禁用（防冲突）。
- **F-SYS-5 局域网 P2P 自更新（P1，v0.27，决议 #166/#181）**：纯内网、零服务器、无中心的点对点自动更新。节点互相已知彼此平台与版本，发现**同平台、版本更高、且声明可作更新源**的在线节点时，提示"内网有新版 vX.Y.Z（来自 某同事）"；用户确认后从该节点拉取同平台同架构安装包（复用文件传输通道，沿用 SHA-256 完整性校验），安装并重启。**谁先升级谁自然成源**——下载过的节点保留包、接力扩散，无"管理员"角色、无人工放包（升级动作本身使其成源）。本期形态：**Windows 安装版（nsis）**安装时自留一份安装器到数据目录、更新时静默装（per-user 免管理员授权）；**Linux deb** 运行态用 `dpkg-deb` 把自身重打包成 deb、更新时经系统授权（pkexec）安装。触发为**发现即提示、用户确认才更新**（不静默）；安全以用户确认 + SHA-256 + 同平台同架构严格匹配 + 包内版本核对 + 内网信任边界（决议 #5）为准。**纯内网零外网，不违反红线 #5**（红线禁的是外网更新检查 / CDN）。mac（dmg/zip）因形态识别与签名 / Gatekeeper 成本暂缓、列后续。分三步落地见 tech-design §12。
- **F-SYS-6 首次启动向导（P0，v0.1，决议 #22）**：首次打开必经：昵称（必填，默认取系统用户名）→ 公司/部门/团队（选填）→ 文件保存目录确认 → 进主界面。避免全网一片"未命名"。

### 6.7 设置

设置页按常见桌面软件的信息架构组织为 8 组（决议 #14/#31）：账号资料、通用外观、消息通知、聊天与文件、网络连接、快捷键、安全与高级、关于。**明细以 [ui-design.md](ui-design.md) §8 为准**。所有设置本地持久化；提供"重置为默认"。

### 6.8 本地开放接口（远期预研，决议 #21）

为未来 AI 集成预留：本机的外部程序可经接口**读取**聊天（会话/消息/联系人查询、新消息订阅）与**接管**聊天（以本人身份发消息、自动回复）。**不进当前任何版本计划**，但约束现在立好：

- **默认完全关闭**，需用户在设置中显式开启；
- 仅监听 `127.0.0.1`（数据不出本机，与"纯内网"原则一致）；
- token 鉴权；**读与写（接管）分级授权**；
- 接口操作写审计日志；AI 代答的消息建议带可见标识（细则届时定）；
- 接口形态（本地 HTTP/WebSocket 或 MCP 服务器）按届时 AI 生态选型；
- 架构按"可挂接"预留，见 tech-design.md §3。

## 7. 非功能需求

| 维度 | 指标 |
|---|---|
| 规模 | 1000 在线节点：列表流畅滚动/搜索，发现协议无广播风暴 |
| 发现时延 | 同网段上线后 ≤ 3s 互相可见；跨网段（gossip）≤ 30s |
| 消息可靠性 | 不静默丢消息：ACK + 重试 + 持久化补发队列；送达状态真实可信 |
| 传输性能 | 千兆内网单流 ≥ 80 MB/s（受写盘限制时以磁盘为准） |
| 资源占用 | 常驻内存目标 ≤ 300 MB，空闲 CPU ≈ 0%，安装包 ≤ 100 MB |
| 数据安全 | 历史库与队列崩溃不损坏（SQLite WAL）；卸载可选择保留/清除数据；本地为明文存储——不做库加密（与决议 #5 同理），依赖系统账户隔离；诊断日志只记元数据、永不记消息正文（决议 #22） |
| 兼容性 | 严格遵守 README「开发红线」（Chrome 108 / Node 16.17 基线、三平台冒烟） |
| 隐私 | 零遥测、零外网请求；不做传输加密，内网明文（决议 #5） |

- 2026-06-26 v1.40 决议 #164：修复 issue #3「发图片 → 接收方收到、发送方显示传输失败」。**根因**：图片走文件传输通道，发送方那条消息的成功/失败状态目前只由 offer 控制报文的回程 ACK 决定，跟图片数据是否真的传完是解耦的；有 UDP 丢包时——对端其实第一次就收到 offer、自动接收并通过 TCP 把整张图拉走（所以接收方一切正常），但 offer 的回程 ACK 丢了，发送方 UDP 重传（1+2+4s）加 TCP 兜底全部判负（7~14s），而局域网图片数据通常 1s 内就传完，迟到的判负把已经传完的这条消息标成了「失败」。**修复**（`files.ts`，纯 services 状态机，不动协议 / 数据面 / 扫描发现）：以数据面真实结果为准——① TCP 供流完成的 `served` 事件 → `applyMsgStatus('sent')`（数据完整送达即已发送）；② 收到对方 `accept` 控制报文 → `applyMsgStatus('sent')`（对方已接受即已送达）；③ `applyMsgStatus` 加终态保护：消息一旦为 `sent`，迟到的 `failed` 不再覆盖（与既有 `recalled` 保护并列）；④ offer 可靠发送回调 `ok=false` 时先查 transfer 状态，已 `accepted`/`done` 就不再 `finish('failed')`、不翻消息失败（单聊 `offerPaths` 与群聊 `offerGroupPaths` 各改一处）。新增 3 个 `files.test.ts` 回归用例（offer ACK 丢失但 served / accept 均判 sent、offer 失败且无送达迹象仍判 failed）。版本 0.26.3 → **0.26.4**（bug 修复，按决议 #73 patch +1）。
